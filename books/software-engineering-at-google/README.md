## Key takeaways

### Chapter 1 - What is software engineering?

- Sustainable software engineering is socio-technical, not only technical.
- The difference between "works now" and "is maintainable" defines real system quality.
- Sustainability depends on changing safely throughout the full code lifecycle.
- Tradeoffs are unavoidable and must be made explicit.

### Chapter 2 - Working well in teams

- Visible collaboration lowers risk and prevents investing time in the wrong direction.
- Humility, respect, and trust are technical preconditions for healthy team scaling.

### Chapter 3 - Sharing knowledge

- Knowledge sharing reduces single points of failure and speeds learning capacity.
- Psychological safety enables questions, living documentation, and sustainable improvement.

### Chapter 9 - Code review

- Well-designed code review protects correctness, consistency, and maintainability.
- Review quality depends on small changes, professional feedback, and explicit criteria.

## Favorite ideas

- Optimize for maintainability over local speed.
- Build organizational learning into everyday workflows.

## Practical applications

### Chapter 1 - What is software engineering?

- Add decision logs for architecture and technical debt tradeoffs.
- Revisit critical technical decisions periodically throughout system lifetime.
- Design scaling policies aligned with actual continuous integration coverage.

### Chapter 2 - Working well in teams

- Formalize blameless post-mortems with concrete changes as required output.
- Encourage early feedback to avoid long rework cycles.

### Chapter 3 - Sharing knowledge

- Institutionalize internal sharing rhythms: weekly tips and technical newsletters.
- Keep documentation searchable and editable to prevent information islands.

### Chapter 9 - Code review

- Standardize reviews around readability, simplicity, and shared learning.
- Automate repetitive validations so human review can focus on design.
- Define chapter-level checklists to turn concepts into team habits.
